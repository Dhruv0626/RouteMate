# 🛡️ RouteMate: Database Schema & Workflow Architecture

This document clarifies the 11 MongoDB collections in RouteMate and provides a deep dive into the synchronization between **PublishedRide** and **Trip** models.

---

## 1. 📂 Core Models (Active Booking Flow)

### 👤 User (`models/User.js`)
- **Use Case**: Identity and Access Management.
- **Working**: Stores credentials, roles (`passenger`, `driver`, `admin`), and personal info. Every other document references this ID.

### 🚗 DriverProfile (`models/DriverProfile.js`)
- **Use Case**: Driver verification and live status.
- **Working**: Stores KYC, vehicle details, and real-time availability/location.

### 🗺️ PublishedRide (`models/PublishedRide.js`)
- **Use Case**: The "Vehicle" or "Pool" state.
- **Working**: Represents a driver's intention to travel a route. It can hold multiple `bookings`. It tracks the overall vehicle status (open, full, arrived, active).

### 📍 Trip (`models/Trip.js`)
- **Use Case**: The "Journey" or "Transaction" state.
- **Working**: Represents a specific passenger's ride from A to B. It is created when a driver accepts a booking. It tracks the passenger's experience (searching, matched, arrived, ongoing, completed).

---

## 2. 🔄 Deep Sync: PublishedRide ↔ Trip

While **PublishedRide** tracks the driver's vehicle, the **Trip** tracks the passenger's journey. They are connected via the `publishedRide` field in the Trip model.

### Lifecycle Status Mapping
The system ensures that when a driver updates their status, all associated passenger trips are synchronized.

| Event | PublishedRide Status | Trip Phase | Description |
| :--- | :--- | :--- | :--- |
| **Driver Posts** | `open` | *N/A* | Driver offers a ride on a specific route. |
| **Request Recv** | `full` | *N/A* | Passenger requests a seat. Booking is `pending`. |
| **Driver Accepts**| `active` | `matched` | Trip created. OTP generated and sent to passenger. |
| **Driver Arrives**| `arrived` | `arrived` | Driver reached pickup. Passenger notified to share OTP. |
| **Trip Starts** | `active` | `ongoing` | Driver enters 4-digit OTP. Trip officially starts. |
| **Trip Ends** | `completed` | `completed` | Driver completes ride. Stats updated. |

## 📈 Graph-wise Working Flow

```mermaid
graph TD
    A[Driver: Publish Ride] -->|status: open| B(PublishedRide Created)
    B --> C[Passenger: Search & Book]
    C -->|status: full| D{Driver: Respond}
    
    D -->|Reject| E[Status: open]
    D -->|Accept| F[Booking Confirmed]
    
    F -->|ride: active| G[Trip Created: matched]
    G --> H[Driver Arrival]
    
    H -->|ride: arrived| I[Trip: arrived]
    I --> J{Enter OTP}
    
    J -->|Verified| K[Trip: ongoing]
    K --> L[Driver: Finish]
    
    L -->|ride: completed| M[Trip: completed]
    M --> N[Earnings & Stats Updated]
```

## 🚀 The Working Flow (Step-by-Step)

1. **Discovery**: Driver publishes a route (`PublishedRide`). Passenger searches using source/destination cities and road-aware proximity.
2. **Booking**: Passenger selects a ride. The **Fare Engine** calculates the "Final Fare" based on road distance. Booking is saved as `pending` inside the `PublishedRide`.
3. **Acceptance**: Driver is notified. Upon "Accept", the system:
    - Sets `PublishedRide` to `active`.
    - Creates a new **Trip** record linked to that ride.
    - Generates a **4-digit OTP** stored in the Trip.
    - Notifies the passenger with the "Booking Confirmed" message + OTP.
4. **Pickup**: Driver navigates to the passenger. When they click "I Have Arrived":
    - Both `PublishedRide` and `Trip` switch to `arrived`.
    - Passenger gets a notification: "Share OTP to start".
5. **On-Trip**: Driver enters the OTP.
    - `Trip.phase` becomes `ongoing`.
    - `Trip.startedAt` is recorded.
6. **Completion**: Driver arrives at the drop-off and clicks "Complete Ride":
    - Both sync to `completed`.
    - Final stats (Trips count, Earnings) are incremented in User and DriverProfile models.
