Here is the complete list of requirements and changes needed to successfully deploy the RouteMate project to Render for stable performance.

### 1. Backend Changes (Node.js/Express)
- **Dynamic Port:** Ensure your `server.js` listens on `process.env.PORT` provided by Render. DO NOT hardcode the port (e.g., `const PORT = process.env.PORT || 5000;`).
- **CORS Configuration:** Update your CORS middleware to explicitly allow requests ONLY from your upcoming Render frontend URL in production, instead of '*'.
- **Start Script:** You already have `"start": "node server.js"` in your `package.json`, which Render will use to start the application.
- **Dependency Management:** Ensure all required modules are listed in `dependencies` (not `devDependencies`). Since `nodemon` is in `devDependencies`, Render will safely ignore it in production.
- **Redis Requirement:** Because your application uses `ioredis` for caching, you must allocate a Render Redis instance (or use a free tier on Upstash) and configure the Redis connection URL in your environment variables. Without this, your app will crash when trying to connect to `localhost:6379`.

### 2. Frontend Changes (Vite/React)
- **API Base URL:** Make sure your frontend fetches data using an environment variable (e.g., `import.meta.env.VITE_API_URL`). Change it from `http://localhost:5000` to the Render backend URL.
- **Build Command:** The command for Render should be `npm install && npm run build`. Render will statically serve the `dist` folder.
- **Static Site Routing (Crucial for SPA):** Since this is a React application using `react-router-dom`, you need to set up a rewrite rule in the Render Dashboard so page refreshes don't return a 404 error:
  - Go to your Static Site settings on Render.
  - Go to the **Redirects/Rewrites** section.
  - Add a rule: Source: `/*`, Destination: `/index.html`, Action: `Rewrite`.

### 3. Environment Variables (To be set on Render Dashboard)

**For Backend (Web Service):**
- `PORT`: (Render will automatically assign this, but good to know)
- `NODE_ENV`: `production`
- `MONGO_URI`: Your MongoDB Atlas connection string.
- `CLOUDINARY_CLOUD_NAME`: Your Cloudinary cloud name.
- `CLOUDINARY_API_KEY`: Your Cloudinary API key.
- `CLOUDINARY_API_SECRET`: Your Cloudinary API secret.
- `REDIS_URL`: Your Render Redis or external Redis (Upstash) connection string.
- `JWT_SECRET`: A secure, strong string for authentication tokens.
- `FRONTEND_URL`: The URL of your deployed Vite frontend (for CORS).

**For Frontend (Static Site):**
- `VITE_API_URL`: The URL of your backend (e.g., `https://routemate-backend.onrender.com`).

### 4. Performance & Stability Recommendations
- **Database Connection:** Use robust MongoDB connection settings with automatic reconnects (`useNewUrlParser`, `useUnifiedTopology` if applicable to Mongoose v9).
- **Cluster/PM2:** By default, Node runs on a single thread. Render dynamically assigns resources. For extreme scale, you could use `pm2` to utilize all CPU cores, but Render's built-in auto-scaling works fine for standard tiers.
- **Health Check Endpoint:** Create a simple `GET /health` endpoint that returns `200 OK`. You can plug this into Render so it knows if your service crashes and automatically restarts it.
- **Rate Limiting Details:** You use `express-rate-limit`. Make sure it uses `app.set('trust proxy', 1);` since Render sits behind a load balancer and reverse proxy; otherwise, rate limiting will block all users mapping them to a single Render proxy IP.

### 5. Third-Party Integrations (OAuth & Maps)
- **Google OAuth:** In the Google Cloud Console, update your "Authorized JavaScript origins" with your frontend Render URL and "Authorized redirect URIs" with your backend callback URL (e.g., `https://<YOUR-RENDER-BACKEND-URL>/api/users/auth/google/callback`).
- **Facebook OAuth:** Go to the Facebook Developer Dashboard for your app. In the Facebook Login Settings, add your new backend callback URL to the "Valid OAuth Redirect URIs" (e.g., `https://<YOUR-RENDER-BACKEND-URL>/api/users/auth/facebook/callback`).
- **OAuth Environment Variables:** Add `GOOGLE_CALLBACK_URL` and `FACEBOOK_CALLBACK_URL` to your Render backend environment variables targeting the new production endpoints.
- **Leaflet JS Map:** Your implementation leverages standard OpenStreetMap tiles out-of-the-box (`https://{s}.tile.openstreetmap.org...`). Because these do not enforce domain whitelisting and are naturally served over HTTPS, **no changes are required at all for Leaflet** â€” the map will work perfectly in production.
